# 0020. Use UUIDv4 for All Application-Generated IDs

**Date**: 2026-05-04

**Status**: proposed

**Deciders**: Theo <!-- project extension, not in Nygard's original -->

**Confidence**: high — the timestamp leakage concern is acknowledged even by UUIDv7 proponents
(Aiven, Marc Brooker / AWS). Verification of our actual stack revealed that the libraries we
already use (Emmett, BullMQ) handle their own internal locality with `BIGSERIAL` and counter-based
IDs, so there is no internal application-generated position where UUIDv7 would buy us anything.
The decision reduces to: UUIDv4 for everything we generate; the libraries handle the rest.

**Reevaluation triggers**: a future module introduces a CRUD table that grows large enough for
insert-locality to become a measurable bottleneck _and_ the table's IDs are guaranteed to never
cross a public boundary (at which point UUIDv7 becomes a localized, per-table decision worth its
own ADR); a public requirement emerges for ID-based ordering visible to clients (creation time
becomes deliberately public per the Twitter Snowflake precedent); a future ADR introduces a
separate human-facing booking confirmation code, alongside the aggregate_id.

## Context

Our current convention lists "uuid v7" as the project-wide ID default. Combined with our
frontend-generated ID pattern (Udi Dahan: client generates the aggregate ID at first user
interaction, enabling optimistic UI and natural command idempotency), every `aggregate_id` is a
UUIDv7 created on the client and reused as:

- The event stream key inside Emmett (ADR-0019)
- The command identity / idempotency key
- The projection primary key in read models
- **The public resource ID exposed in URLs (`/api/listings/{id}`, `/api/bookings/{id}`), API responses, and likely invoice references**

UUIDv7 encodes a 48-bit Unix millisecond timestamp in its most significant bits. This is the
feature that gives it B-tree insert locality. It is also a metadata leak the moment the ID is
exposed at any public boundary. A scraper of public listing URLs can reconstruct Parko's listing
growth curve; sequential gaps in booking IDs reveal booking volume per time window; invoice
references issued for DAC7 reporting expose ordering and throughput. Marc Brooker (AWS) and
Aiven engineering both explicitly recommend UUIDv7 only for internal keys, with a separate
UUIDv4 for external identifiers.[^1][^2]

Verification of our actual stack revealed that the natural homes for UUIDv7 are already taken
by library-managed counters, not application-generated UUIDs:

- **Emmett's PostgreSQL event store** uses `bigint` for both `StreamPosition` and `GlobalPosition`
  (verified in Emmett's source), corresponding to a `BIGSERIAL` primary key — the same pattern
  Message-DB and Eventuous use. Insert locality on the event store is already optimal; it is
  not a position we generate IDs for.
- **BullMQ job IDs** default to an auto-incrementing counter scoped per queue, not UUIDs.
  Custom job IDs are a deduplication tool; in our stack the natural value to reuse there is the
  existing aggregate_id, which is already governed by this ADR.
- **OpenTelemetry** generates its own `traceId` / `spanId` formats — not UUIDs at all.

The "internal-only UUIDv7" carve-out has no concrete consumer in our application code. The
decision therefore simplifies to a single rule: UUIDv4 for everything we generate.

For Parko specifically:

- We are at MVP scale. UUIDv7's insert-locality benefit is irrelevant for our generated IDs
  because we don't generate IDs at any position where insert locality matters.
- We are a P2P parking marketplace. Listing volume, booking volume, and host activity windows
  are competitively sensitive — these are exactly the leaks UUIDv7 produces at public boundaries.
- French regulatory artifacts (TVA invoices, DAC7 reports, the three-invoice mandate) will
  reference IDs in ways that may be visible to authorities, hosts, guests, and tax processors.
- GDPR posture: timestamp metadata in IDs is a correlation vector, in tension with our
  "PII out of event payloads from day one" stance.

This decision builds on the Udi Dahan frontend-generated ID pattern (project knowledge:
udi-dahan-decouple-services) and ADR-0019 (Emmett as event store infrastructure).

## Decision

**We will use UUIDv4 for every ID generated by application code. Internal infrastructure
positions managed by libraries (Emmett event-row PK, BullMQ job counter, OpenTelemetry trace
IDs) are not application concerns and use whatever scheme the library defaults to.**

```
ID STRATEGY — Parko

APPLICATION-GENERATED IDS (always UUIDv4):
  ✅ aggregate_id (Booking, Listing, Payment, Discovery)        → UUIDv4
  ✅ Command idempotency key (same value as aggregate_id)       → UUIDv4
  ✅ Read-model projection primary key                          → UUIDv4
  ✅ Any DTO field crossing the API boundary                    → UUIDv4
  ✅ Custom BullMQ job IDs when used for deduplication          → reuse the aggregate_id (UUIDv4)

LIBRARY-MANAGED INTERNAL IDS (we do not generate these):
  • Emmett event-row PK / global_position / stream_position     → BIGSERIAL (Emmett owns this)
  • BullMQ default job IDs                                      → auto-incrementing counter (BullMQ owns this)
  • OpenTelemetry trace_id / span_id                            → OTel-generated (W3C trace context)

FUTURE — HUMAN-FACING IDENTIFIERS:
  ✅ Booking confirmation codes shown in UI ("BK-7HJ4P2")       → structured short code
     (projected into read models alongside the aggregate_id, not a replacement for it)

REJECTED FOR PARKO:
  ❌ UUIDv7 as the application-wide default
     (no application-generated position benefits from its locality;
      every application-generated ID has a public-exposure path)
  ❌ NanoID for frontend-generated aggregate IDs
     (Postgres `uuid` columns + Emmett event store schema expect standard UUID format)
  ❌ BIGINT auto-increment as application-generated primary key
     (incompatible with frontend-generated ID pattern; requires DB authority)

NEVER:
  ❌ Use any UUID variant as a security token or authorization secret
     (UUIDs are addresses, not capabilities — authorization belongs in
      session/scope/ownership checks per ADR for IDOR-style protection)
```

Tooling: replace `uuid v7` generator calls with `uuid v4` at the frontend ID-provider port.
The `IdProvider` port in `src/core/id-provider/` keeps its current shape; only the concrete
implementation changes. Test fakes (`SequentialIdProvider`) are unaffected.

Migration: there is nothing in production yet. New code uses UUIDv4 immediately. Any existing
generated test fixtures or seed data are regenerated.

## Consequences

### Positive

- Public URLs (`/api/listings/{id}`, `/api/bookings/{id}`) no longer leak creation timestamps,
  ordering, or business volume to scrapers, competitors, or authorities.
- DAC7 / invoice references issued under the French regulatory framework no longer carry
  embedded timing metadata.
- Reduces friction with the "PII out of event payloads from day one" posture — IDs themselves
  stop being a correlation vector.
- The Udi Dahan frontend-generated ID contract is preserved unchanged (UUIDv4 satisfies the
  same idempotency-key requirement as UUIDv7).
- Single, unconditional rule for contributors: "use UUIDv4." No mental overhead of deciding
  whether a position is "internal enough" for UUIDv7.

### Negative

- We close the door on UUIDv7 for any future application-generated ID without a follow-up ADR.
  If a future module introduces a CRUD table that grows large enough for insert locality to
  matter _and_ whose IDs never cross a public boundary, that table needs its own decision.
- If we later introduce human-facing booking codes, two ID concepts will be in play —
  the aggregate_id (UUIDv4) and the human-facing public_code (structured) — and read-model
  projections will need to project both. This is not unique to this ADR but is the natural
  consequence of separating addressing from human-facing display.

### Neutral

- Storage size is identical (UUIDv4 and UUIDv7 are both 128 bits). The change is invisible to
  the database wire format.
- Generation performance is comparable; both are O(1) on the client.
- No locality is given up by application code, because Emmett's event store already uses
  `BIGSERIAL` internally — that locality was never ours to give up or keep.

## Alternatives Considered

### Alternative 1: Keep UUIDv7 everywhere (status quo)

Rejected. The post-launch cost of fixing this — adding a `public_id` column projected onto
every public-facing read model, building backward-compatibility shims for any URL already in
the wild, dealing with migration of cached SEO content — is significantly higher than the
near-zero cost of switching now while no production data exists. The insert-locality benefit
UUIDv7 provides is not yet a realized win at our scale, so the trade is asymmetric in favor
of switching.

### Alternative 2: Dual-ID with BIGINT internal + UUIDv4 public (Junco's "boring solution")

Rejected for Parko. The dual-ID pattern with BIGINT requires the database to own ID
generation, which conflicts directly with the Udi Dahan frontend-generated ID pattern we
have committed to. We would lose the optimistic UI / command idempotency benefits to gain
locality we do not yet need. If we were a CRUD-first system without frontend-generated IDs,
this would be the right call.

### Alternative 3: NanoID for public-facing IDs

Rejected as the _aggregate_id_ choice. NanoID does not produce a standard UUID format, which
breaks compatibility with Postgres `uuid` columns and the Emmett event store schema. NanoID
remains a candidate for a future _human-facing_ short-code (booking confirmation code visible
in the UI), where compactness matters and the value is decoupled from the aggregate_id.

### Alternative 4: ULID or KSUID for any future internal-only need

Considered for future reference. ULID and KSUID share UUIDv7's timestamp-prefix design and the
same leakage problem when exposed publicly. ULID has marginally better collision properties[^3].
None of these are needed today because we have no application-generated position where insert
locality is a concern. If a future CRUD module surfaces such a position, that ADR can pick
between UUIDv7, ULID, and KSUID on its own merits — not as a project-wide default.
